# Документационный workflow
## Цель
Обновлять документацию PESKOVP предсказуемо, безопасно и с контролем качества.

## Базовые правила
- Работать через отдельную ветку от `master`.
- Делать атомарные изменения: один смысловой блок на один commit.
- Не добавлять секреты (`API keys`, токены, приватные URL/пути).
- Проверять, что ссылки и пути в документации соответствуют текущей структуре проекта.

## Минимальный checklist перед PR
1. Обновлены целевые doc-файлы.
2. Проверена консистентность с:
   - `README.md`
   - `TODO_PLAN.md`
   - актуальными `reports/*.md`
   - `docs/README.md` (индекс и ссылки на разделы)
3. Нет утечек секретов.
4. Изменения описаны в PR кратко и предметно.
5. Для operational ограничений добавлен явный статус (`active`/`waived`/`resolved`) и критерий закрытия.

## Рекомендуемый формат PR для docs
- **Summary**: какие разделы обновлены.
- **Reason**: зачем изменения нужны именно сейчас.
- **Validation**: как проверялась корректность (paths/ссылки/фазовый контекст).

## Публикация
После merge в `master` — при необходимости синхронизировать связанные phase-отчёты.
## Рекомендуемая структура коммитов для Phase 14
- `docs(index): ...` — обновления `docs/README.md`.
- `docs(api): ...` — API контракты и примеры.
- `docs(runbook): ...` — operational инструкции и диагностика.
- `docs(ops): ...` — waivers, ограничения, и критерии их снятия.
